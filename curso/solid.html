A origem

Os princípios foram reunidos por Robert C. Martin em “Design Principles and Design Patterns” (2000); a ordem que forma a sigla SOLID é de Michael Feathers.

A grande ideia

“O único jeito de ir rápido é ir bem.”

— Robert C. Martin (“Uncle Bob”), Clean Architecture (2017) · tradução livre. SOLID existe para uma coisa: gerir as dependências entre módulos antes que elas giram você.

Os sintomas do design que apodrece — rigidez, fragilidade, imobilidade e viscosidade — são de Robert C. Martin, “Design Principles and Design Patterns” (2000).

As cinco letras

S · Responsabilidade única

“Uma classe deve ter um, e apenas um, motivo para mudar.” — Robert C. Martin, Agile Software Development: Principles, Patterns, and Practices (2002) · tradução livre.

S, em código

Três motivos para mudar

classe RelatórioDeVendas
  método calcularTotais()
  método formatarHtml()
  método gravarNoBanco()

// finanças, design e dba
// mexem na mesma classe —
// e um quebra o outro

Um motivo por classe

classe CalculadoraDeTotais
  método calcular(vendas)

classe FormatadorDeHtml
  método formatar(totais)

classe RepositórioDeVendas
  método gravar(relatório)

O · Aberto/fechado

“Entidades de software devem ser abertas para extensão, mas fechadas para modificação.” — Bertrand Meyer, Object-Oriented Software Construction (1988) · tradução livre.

O, em código

Modificação · reabre o que funciona

método desconto(pedido, tipo)
  se tipo = "natal"
    retorna pedido.total · 0,10
  senão se tipo = "cyber"
    retorna pedido.total · 0,25
  senão se …

// cada campanha nova
// reabre esta função

Extensão · só adiciona

interface Promoção
  método desconto(pedido)

classe PromoçãoDeNatal
  implementa Promoção
  método desconto(pedido)
    retorna pedido.total · 0,10

// campanha nova = classe nova

L · Substituição de Liskov

“Objetos de um subtipo devem poder substituir objetos do tipo-base sem alterar as propriedades do programa.” — Barbara Liskov e Jeannette Wing, “A Behavioral Notion of Subtyping” (1994) · tradução livre.

L, em código

Parece é-um…

classe Quadrado herda de Retângulo
  método definirAltura(v)
    altura  = v
    largura = v            // surpresa

r = novo Quadrado()
r.definirLargura(4)
r.definirAltura(5)
r.área()                   // 25, não 20

…mas não se comporta como um

interface Forma
  método área()

classe Retângulo implementa Forma
  privado largura, altura

classe Quadrado implementa Forma
  privado lado

// irmãos: nenhum contrato quebrado

I · Segregação de interfaces

“Clientes não devem ser forçados a depender de interfaces que não usam.” — Robert C. Martin, Agile Software Development: Principles, Patterns, and Practices (2002) · tradução livre.

I, em código

Interface gorda

interface Máquina
  método imprimir(doc)
  método digitalizar(doc)
  método grampear(doc)

classe ImpressoraSimples
  implementa Máquina

  método grampear(doc)
    falha "não suportado"   // cheiro

Interfaces enxutas

interface Impressora
  método imprimir(doc)

interface Grampeadora
  método grampear(doc)

classe ImpressoraSimples
  implementa Impressora    // só o que usa

D · Inversão de dependência

“Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações.” — Robert C. Martin, “Design Principles and Design Patterns” (2000) · tradução livre.

D, em código

Dependência direta

classe Pedido
  privado banco = novo BancoSql()

  método fechar()
    banco.executar("insert …")

// trocar o banco (ou testar)
// = reescrever a regra

Dependência invertida

interface Repositório
  método salvar(pedido)

classe Pedido
  privado repo: Repositório

  método fechar()
    repo.salvar(isto)

// o detalhe é que se ajusta

Princípios, não leis

“A duplicação é bem mais barata que a abstração errada.”

— Sandi Metz, “The Wrong Abstraction” (2016) · tradução livre

  • SOLID gerencia dependências — não é checklist de cartório
  • Abstração especulativa custa caro: espere a segunda razão real
  • YAGNI e simplicidade também são princípios
  • O sênior sabe quando um princípio não se aplica

Vamos praticar?

  • Ache uma classe com dois motivos de mudança — e separe os atores
  • Troque uma cadeia de se/senão de regras por implementações de um contrato
  • Cace um “novo” de infraestrutura dentro de regra de negócio e inverta a seta
