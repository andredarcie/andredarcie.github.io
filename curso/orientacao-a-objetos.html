A origem

O termo “orientação a objetos” é de Alan Kay, inspirado em biologia: células independentes que cooperam trocando sinais.

A grande ideia

“Lamento ter cunhado, há tanto tempo, o termo ‘objetos’, porque ele leva muita gente a focar na ideia menor. A grande ideia é mensagens.”

— Alan Kay, criador do Smalltalk e do termo “orientação a objetos” · e-mail à lista Squeak, 1998 · tradução livre

Mensagens, em código

// orientação a objetos não é chamar funções sobre dados —
// é enviar mensagens e deixar o objeto decidir

impressora.imprimir(relatório)

// como imprimir é segredo da impressora;
// quem pede só conhece a mensagem

O que é um objeto

“Um objeto tem estado, comportamento e identidade.” — Grady Booch, Object-Oriented Analysis and Design with Applications (1991) · tradução livre.

O objeto, em código

classe Pedido                          // identidade

  privado itens  = []                  // estado
  privado status = ABERTO

  método adicionar(item)               // comportamento
    se status ≠ ABERTO então falha "pedido fechado"
    itens.acrescenta(item)

  método fechar()
    se itens.vazio então falha "pedido sem itens"
    status = FECHADO

Os quatro pilares

Encapsulamento

01

O estado é privado. O mundo conversa com o objeto pela interface pública — e só por ela.

Abstração

02

Expor o essencial, esconder o detalhe. Quem usa conhece o contrato, não a implementação.

Herança

03

Uma classe nasce de outra e ganha sua estrutura. Poderosa — e a mais fácil de usar errado.

Polimorfismo

04

A mesma mensagem, respostas diferentes. Cada objeto responde do seu próprio jeito.

Encapsulamento

Ocultação de informação: “cada módulo esconde uma decisão de projeto que pode mudar” — David L. Parnas, “On the Criteria to Be Used in Decomposing Systems into Modules” (1972) · tradução livre.

Encapsulamento, em código

Estado exposto

conta.saldo = conta.saldo - 100

// compila e roda — e nada garante nada:
// qualquer código, de qualquer lugar,
// pode deixar o saldo inválido

Estado protegido

classe Conta
  privado saldo

  método sacar(valor)
    se valor > saldo
      então falha "saldo insuficiente"
    saldo = saldo - valor

Abstração

“Programe para uma interface, não para uma implementação.” — Gamma, Helm, Johnson e Vlissides, Design Patterns (1994) · tradução livre.

Abstração, em código

interface Notificador
  método notificar(mensagem)

classe Pedido
  privado notificador: Notificador     // o contrato — não a implementação

  método fechar()
    …
    notificador.notificar("pedido fechado")

pedido = novo Pedido(novo PorEmail())   // qualquer implementação serve

Herança

  • Reuso máximo, acoplamento máximo: o par vem junto
  • Só quando é “é-um” de verdade: todo círculo é uma forma
  • Classe-base frágil: mudou o pai, quebraram os filhos
  • Prefira herdar de abstrações a herdar de concretos

“Herança é, no fundo, um mecanismo de delegação automática de mensagens.” — Sandi Metz, Practical Object-Oriented Design (2012) · tradução livre.

Herança, em código

classe abstrata Forma
  método abstrato área()

classe Círculo herda de Forma         // todo círculo É uma forma
  privado raio

  método área()
    retorna π · raio²

// se a frase “X é um Y” soa falsa, não herde

Polimorfismo

Princípio da substituição: todo subtipo deve poder substituir seu tipo-base sem quebrar o programa — Barbara Liskov, “Data Abstraction and Hierarchy” (1987) · Turing Award 2008.

Polimorfismo, em código

formas = [ novo Círculo(2), novo Retângulo(3, 4), novo Triângulo(3, 4) ]

para cada forma em formas
  imprime forma.área()                 // a mesma mensagem…

// saída: 12,56…   12   6              — três respostas diferentes

Composição sobre herança

“Prefira composição de objetos a herança de classes.” — GoF, Design Patterns (1994). “Você queria uma banana, mas ganhou um gorila segurando a banana — e a floresta inteira.” — Joe Armstrong, sobre herança, em Coders at Work (2009) · traduções livres.

Composição, em código

Herança · uma classe por combinação

classe Forma
classe FormaComBorda
  herda de Forma
classe FormaComBordaESombra
  herda de FormaComBorda

// e a forma com sombra, sem borda?
// mais uma classe — para sempre

Composição · peças combinadas

classe Forma
  privado efeitos = []

  método com(efeito)
    efeitos.acrescenta(efeito)
    retorna isto

forma.com(borda).com(sombra)

Diga, não pergunte

Tell, Don’t Ask — Andy Hunt e Dave Thomas (The Pragmatic Programmer, 1999). O “modelo de domínio anêmico” é anti-padrão — Martin Fowler (2003).

“Diga”, em código

Pergunte · a regra vaza

// no serviço
saldo  = conta.saldo()
limite = conta.limite()
se saldo + limite ≥ 100
  conta.definirSaldo(saldo - 100)

Diga · a regra mora na conta

// no serviço
conta.sacar(100)

// a decisão inteira acontece
// dentro do objeto
